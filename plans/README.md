## Findings considered and rejected

- **Migration `SAVEPOINT` never rolled back on failure, "connection handed back in a corrupted state"** (originally raised as a correctness finding): vetted by tracing the actual call path — `run_migrations(...)?` is called inside `SqlitePool::open`, so a migration failure propagates immediately via `?` and `open()` returns `Err` before the connection is ever reused; SQLite auto-rolls-back an unreleased savepoint when a connection closes without committing it. The described persistent-corruption scenario does not occur in the current code. Folded into plan 005 (Step 6) as a test that *verifies* this is true rather than a standalone fix — if that test ever fails, it should become its own finding at that point, not before.
- **SSRF hardening test for the (currently hardcoded, unexploitable) OpenRouter base URL**: real but low-value while only one provider exists and the URL is a compile-time constant with no IPC-controllable input. Worth a one-line regression test if/when a second provider or a user-configurable base URL is ever introduced — not before.
- **`.env.example` missing for the `OPENROUTER_API_KEY` dev-only env fallback**: real onboarding friction, but minor enough not to warrant a standalone plan; whoever picks up plan 005 or 008 next should add a one-line `.env.example` as a drive-by while in that area, rather than spending a separate plan-and-review cycle on it.
- **`SqlitePool::with_conn` panics (rather than returning an error) on a poisoned mutex**: confirmed as a deliberate, commented design choice (`storage/sqlite.rs:61-66`'s own comment: "propagate the panic rather than misrepresenting a fatal concurrency failure as a parameter error"). Flagged during audit as a severity-mismatch worth a second look (one panic anywhere in storage code currently takes down all future storage calls for the app session), but changing fail-fast-to-recoverable is a judgment call for the team, not a clear bug fix — not planned as a fix; left as a documented open question for a future architecture discussion.
- **`ipc/chat.rs` trending toward a "god module"** (723 lines, mixing IPC contract + streaming orchestration + attachment handling): a real architectural trend, not a present violation of the "thin bootstrap" rule. Worth revisiting before the next major chat feature lands, not urgent enough for a standalone plan today.
