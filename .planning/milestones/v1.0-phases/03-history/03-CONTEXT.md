# Phase 3: History - Context

**Gathered:** 2026-06-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Persist and search local conversation history with recoverable storage behavior. This phase implements:

- The `conversations` and `messages` tables in SQLite, with migrations
- FTS5 full-text search over message content (external-content table + triggers)
- The `ipc/history.rs` command surface (list, get, delete, search)
- The History surface frontend (scrollable conversation list + search bar)
- Wiring `chat_send`'s `conversation_id` parameter to actual storage writes
- Manual conversation delete with WAL checkpoint after deletion

This phase does NOT include: auto-expiry retention rules (future settings phase), export/import backup (future), conversation renaming, artifact reconstruction, or settings UI.

</domain>

<decisions>
## Implementation Decisions

### Conversation Schema

- **D-01:** Two-table model: `conversations(id, title, model, status, created_at, updated_at)` and `messages(id, conversation_id FK, role, content, status, created_at)`. Messages have `ON DELETE CASCADE` from conversations. No `message_count` column — derive via `COUNT` query.
- **D-02:** `conversations.status` values: `'active'` (in-flight or newly created), `'complete'` (stream ended with `ChatEvent::Done`), `'incomplete'` (cancelled — maps to Phase 2 D-06 `status: "incomplete"` decision).
- **D-03:** Conversation title auto-generated by the backend from the first user message, truncated to ~60 characters. User cannot rename in this phase. Title generation is backend-owned and invisible to IPC (not accepted from the frontend).
- **D-04:** Message content stored verbatim in SQLite — full-fidelity required for FTS5 search (HIST-02). Redaction applies to egress surfaces only (logs, telemetry, crash reports, exports). The IPC layer + typed backend commands are the privacy boundary; the renderer never reads the DB directly.
- **D-05:** Model stored on the `conversations` row, sourced from `ChatEvent::Done { model }` (already returned by Phase 2). Written by the backend when the stream completes.

### FTS5 Search

- **D-06:** External-content FTS5 virtual table indexed against `messages.content`. Search scope: message content only (not conversation titles). FTS5 triggers on `messages` INSERT/UPDATE/DELETE keep the index in sync.
- **D-07:** Search triggered in real-time with debounce (~300ms). Frontend invokes `history_search` IPC command as user types; results update live.
- **D-08:** Search results returned at conversation granularity: each result includes the conversation's title, model, status, timestamp, and a short (~80 chars) highlighted snippet from the matching message content. Grouped one row per conversation (multiple matches within a conversation = one row).

### History Surface UX

- **D-09:** History surface layout: scrollable conversation list with a search bar at the top. Each row shows title, model, status, and timestamp. No date-grouping headers in this phase.
- **D-10:** Clicking a conversation in History navigates to the Chat surface and loads that conversation's full message history (all prior messages rendered, same visual treatment as a live session). Input field pre-focused. No continuation banner.
- **D-11:** New conversation: a dedicated "New chat" button in the Chat surface header (and/or the nav rail). Clears the active conversation and creates a fresh `conversation_id`. Previous conversation remains persisted.

### Retention & Deletion

- **D-12:** Phase 3 retention scope: manual delete only. User explicitly deletes individual conversations from the History surface (per-row delete action). No auto-expiry, no background retention jobs in this phase.
- **D-13:** Hard delete: `DELETE FROM conversations WHERE id = ?` with `ON DELETE CASCADE` propagating to `messages`. FTS5 delete triggers handle index cleanup automatically. No soft-delete tombstone column.
- **D-14:** After a hard delete, trigger a WAL checkpoint (`PRAGMA wal_checkpoint(TRUNCATE)`) to reclaim disk space and ensure no unmanaged WAL state. Required by the adversarial hardening spec's WAL lifecycle rule.

### Claude's Discretion

- Exact FTS5 tokenizer configuration (unicode61 vs. ascii — pick based on FTS5 defaults for English content)
- SQL index selection beyond the FK index on `messages.conversation_id` (let query analysis guide)
- Exact IPC command names for the `history_*` surface (e.g., `history_list`, `history_get`, `history_delete`, `history_search`) — follow the naming pattern from `ipc::app_shell`
- WAL checkpoint error handling: if checkpoint returns busy readers, surface as a non-fatal warning (do not block the delete response)

</decisions>

<code_context>

## Existing Code Insights

### Reusable Assets

- `src-tauri/src/storage/sqlite.rs` — `SqlitePool::with_conn()` and `SqlitePool::from_connection()` patterns. New `ConversationStore` and `MessageStore` must use `with_conn()` — never call `with_conn` directly from IPC handlers.
- `src-tauri/src/ipc/app_shell.rs` — `assert_main_window()` helper; `ShellError` enum with `thiserror` + `serde` derive pattern. Copy for `HistoryError`.
- `src/lib/stores/surface.ts` — `normalizeIpcError()` for frontend IPC error normalization. History store should use the same pattern.
- `src-tauri/src/storage/migrations.rs` — `MIGRATIONS` static slice; append new entries for `conversations`, `messages`, FTS5 virtual table, and triggers.

### Established Patterns

- **Typed domain stores:** `ShellPreferenceStore` is the canonical pattern. New stores (`ConversationStore`, `MessageStore`) wrap `SqlitePool` and expose domain-specific methods — no raw `with_conn` in IPC handlers.
- **IPC error shape:** `{ code: "SCREAMING_SNAKE_CASE", message: string }` — enforced at IPC layer via `thiserror` + `serde`. `HistoryError` must serialize to the same shape.
- **Window-label enforcement:** Every history command validates caller via `assert_main_window`.
- **Migration ordering:** Append-only, strictly ascending `id`. New migrations are `"0002"`, `"0003"`, etc.
- **Lock ordering:** Shell lock before SQLite lock (documented in ARCHITECTURE.md). History commands that touch both must maintain this order.

### Integration Points

- `src-tauri/src/main.rs` — `tauri::generate_handler![]` must be updated to register new `history_*` commands. `src-tauri/capabilities/main.json` must be updated with allow grants.
- `src-tauri/src/ipc/chat.rs` (Phase 2 output) — `chat_send` has `conversation_id: Option<Uuid>` wired. Phase 3 implements the actual storage write in the `chat_send` handler, calling into `ConversationStore`/`MessageStore`.
- `src/routes/` — History surface route (`/history` or the history surface view) needs a Svelte component: conversation list, search bar, click-to-continue wiring.

<specifics>
## Specific Ideas

- The user explicitly cited the adversarial hardening spec (`docs/Tauri_Svelte_AI_App_Architecture_Adversarial_Hardened_v5.md`) as the authority for WAL lifecycle requirements. Downstream agents must read it before implementing `retention.rs` and the delete flow.
- WAL checkpoint after delete should use `PRAGMA wal_checkpoint(TRUNCATE)` — ensures WAL file is truncated and disk space reclaimed. Non-fatal if busy readers are present (log a warning, don't block the delete response to the frontend).
- FTS5 external-content table setup: the architecture described an external-content FTS5 table tied to `messages.id` with triggers on inserts/updates/deletes. This is the correct pattern (not a content= shadow table) — lets the main `messages` table remain the source of truth.
- Message snippet in search results: ~80 chars around the match, with the matched term highlighted. FTS5's `snippet()` or `highlight()` auxiliary function can produce this directly from the query.
- The History surface list row should show: conversation title, model badge (e.g., "claude-sonnet-4-6"), status indicator (complete/incomplete), and relative timestamp ("2 days ago"). Same visual density as the Claude sidebar.

</specifics>

<deferred>
## Deferred Ideas

- **Auto-expiry / scheduled retention:** Delete conversations older than N days on a configurable schedule. Belongs in a future settings phase when `retention.rs` gets a scheduling hook.
- **Soft delete / undo:** Tombstone pattern with a recovery window. Not needed for Phase 3 — hard delete is explicit and the WAL provides a short recovery window via checkpoint timing.
- **Conversation renaming:** User edits the auto-generated title. Future phase (needs settings or inline edit UX).
- **Export / import backup:** User-facing backup of the SQLite DB or export to JSON. `backup.rs` scaffold left for this future capability.
- **Title search in FTS5:** Indexing conversation titles alongside message content. Can be added in a future phase if users need it.

None — discussion stayed within phase scope.
