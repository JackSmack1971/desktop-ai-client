<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| human operator → SQLite file | A human runs ad-hoc read-only SQL directly against the app database for inspection (D-04). The query text is human-authored, not renderer-supplied; it crosses no IPC boundary. |
| stored memory rows → human eyes | memory_* table contents become visible to whoever runs the documented queries. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | documented inspection queries (Task 2) | mitigate | Queries use explicit memory_*-only column lists, no joins to messages/conversations/attachments/file_tokens, no SELECT *; acceptance criteria grep-enforce the absence of non-memory_* table reads (D-05). |
| T-02-02 | Tampering | inspection access to the live DB | mitigate | Doc instructs opening the SQLite file in read-only mode so inspection cannot mutate candidate state, decision ledger, or run traces. |
| T-02-03 | Elevation of Privilege | shadow-mode isolation | accept→verify | No IPC command or renderer surface is added (D-04); shadow-mode invariant (no `memory::` use in src-tauri/src/ipc/) is preserved. Rationale: inspection is operator-local direct SQL, not a new program-reachable path. |
| T-02-04 | Information Disclosure | tags column free-form content | accept | Tags are caller-supplied derived strings stored alongside existing derived summaries; no new secret/path class is introduced. The same conversation-cascade retention (privacy-boundaries.md) covers the tags column automatically. |
| T-02-SC | Tampering | npm/pip/cargo installs | mitigate | No new package installs are planned. If serde_json is NOT already in src-tauri/Cargo.toml, the executor hand-encodes the JSON array instead of adding a dependency — no new dependency is introduced by this phase. |
</threat_model>
